Плюсы микросервисов:

1. Гибкая масштабируемость: Микросервисы могут масштабироваться независимо друг от друга.
2. Отказоустойчивость: Если один микросервис падает, это не означает, что всё приложение упадёт.
3. Разные технологии: Могут быть написаны на разных языках программирования.
4. Разработка и тестирование по отдельности: Каждый сервис можно разрабатывать и тестировать независимо.
5. Внедрение по отдельности: Новые функции и исправления могут быть быстро внедрены по отдельности.
6. Single Responsibility: На уровне сервиса.

Минусы микросервисов:

1. Сложность управления и развёртывания: Настройка, мониторинг, логирование, обслуживание и версионирование множества микросервисов может быть сложной задачей.
2. Сложность транзакций: Осуществление транзакций между микросервисами сложнo
3. Сетевые затраты: Cетевые задержки и проблемы с производительностью.
4. Проблемы безопасности: Больше точек входа может означать больше потенциальных угроз безопасности.
5. Дублирование кода: Некоторые функции могут повторяться в разных микросервисах.

Шаблоны микросервисов:

1. API Gateway: Сервис - единая точка входа внешнего API, Делает аутентификацию. Кеширует, агрегирует и делегирует запросы к соответствующим микросервисам.
2. Service Discovery: Предоставляет центральное хранилище для поиска микросервисов по имени.
3. Retry:
4. Circuit Breaker: Отслеживает состояние сервиса и, если он упал, перестаёт отправлять запросы до тех пор, пока сервис не восстановится.
Состояния: Closed(всё ок), Open, Half-Open.
5. Sidecar:Позволяет расширить и улучшить один микросервис без модификации его кода путем добавления дополнительного контейнера в тот же pod.
6. Saga:Для управления транзакциями.
Локальные транзакции в каждом сервисе выполняются, и выдают событие для следующего сервиса, чтобы начать транзакцию. Если какая-либо из транзакций терпит неудачу, выполняются транзакции, отменяющие предыдущую транзакцию.
7. Event Sourcing: Хранение ИЗМЕНЕНИЙ в состоянии приложения как последовательность событий.
8. CQRS (Command Query Responsibility Segregation): Разделение модели чтения данных от модели записи для улучшения производительности и масштабируемости.
9. Backends for Frontends (BFF): Отдельный сервис для каждого интерфейса пользовательского приложения или "фронтенда"
